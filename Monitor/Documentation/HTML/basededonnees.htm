Base de données

 

Généralités

 

La base de données de Monitor a été mise à jour selon la base de données créée pour Webmessenger.

 

Tables impliquées à la génération d’un modèle depuis Conceptor

 

 

pmodel

•         Informations générales du modèle

•         Modelid : identifiant du modèle

•         Modelname : nom du modèle

•         Moderef : code de référence unique du modèle (multi-version)

•         Versionno : numéro de la version

•         Pubdate : date de publication du modèle

 

pattribdef

•         La table pattribdef correspond à chaque attribut existant dans Conceptor (nom, description…). Il s’agit de champs dynamiques.

•         Pattribdef est la clé primaire mais des regroupement sont fait selon les catégories d’attributs (général = 131072x, lien externes = 2097152x)

•         Pattribdef contient également la définition des attributs dynamiques (globaux ou locaux, pas de différence)

•         Attention : enregistrements fixe : exemple 1310721 -> JSP messenger.

 

ppdattr

•         La table ppdattr est une associative entre pattribdef et pmodel. Elle comprend les valeurs saisies pour chaque attributs dans Conceptor

•         Un élément (point de départ, point d’arrivée, processus) du modèle est défini par plusieurs enregistrements de ppdattr regroupés par le numéro commun du champs pdattribid. Il devient uniquement avec le champs prow

 

pdeliv

•         Delivid : clé primaire

•         Modelid : clé étrangère du modèle

•         Procidsrc : clé étrangère (pdattribid) sur la table ppdattr qui représente le processus source

•         Procidtrg : clé étrangère (pdattribid) sur la table ppdattr qui représente le processus destination

•         Pdattribid : clé étrangère sur la table ppdattr qui décrit les informations du délivrable

•         Delivname : nom du livrable (info déjà existante dans la table ppdattr (par le pdattribid)

•         Lateral : flag indiquant si livrable latéral

•         Lateral direction : si lateral = 1, indique la direction (4 possibles)

•         Doublesign : ?

 

pproc

•         pprocid : identifiant de la procédure

•         Wkgid : unité liée

•         Pdattribid : clé étrangère sur les informations de la procédure

•         Modelid : clé étrangère sur le modèle

•         Execid : ? (tout à -1)

•         Proctype : type de la procédure

–        0 : start

–        1 : sortie  

–        2 : normale

•         Doublesign : ? (tout à 0)

 

Table affectées à la génération

 

Attributs dynamiques

•         Les champs des attributs dynamiques lors de la génération sont ajoutés à la structure dans la table pattribdef

•         BUG : les valeurs des attributs dynamiques saisis depuis conceptor ne sont pas rapatriées dans la base Messenger. Ceux-ci devraient se trouver dans la table pinput

 

 

Création d’un nouveau cas

•         Tables touchées

–        Proctrans +1

–        Ptrans +1

–        Pcase +1

 

Pcase

•         Identifie un cas démarré

•         Case id : identifiant unique (incrémentiel)

•         Txindate : date de création

•         Casename : nom du cas en format texte

 

Ptrans

•         La table ptrans comprend une transaction pour un cas donné. Elle comprend d’ailleurs la description donnée à la création du cas.

•         En créant un nouveau processus, et en liant à un sous-cas existant, la transaction va être créée et le caseid conservé

•         Caseid : identifiant du cas

•         Transid : identifiant de la transaction. Généré séquentiellement

•         Parenttransid : référence pour la transaction parente

•         Transname : nom de la transaction

•         Globalyn : ? Toujours à zéro

•         Attribid : ? Non utilisé

 

Pstatus

•         Statusid : identitifant de la table

•         Statusname : valeur texte du status

–        Disponible

–        Bloqué

–        Attente

–        Fini

 

Pproctrans

•         Contenu de la boîte de travail

•         Le pstatus défini l’état selon la table pstatus selon le status

•         Il s’agit du lien entre la procédure, cas et transaction.

•         Cette table est alimentée selon le dernier livrable validé qui connaît la «procédure » suivante (procidtrg)

 

Pproctranstxt

•         Lorsque l’on utilise un « remise au chef » ou « retour pour non-conformité », l’état de la base change pour annuler le dernier « pproctrans » pour revenir à l’état précédent. La table pproctranstxt contiendra le message saisi par l’utilisateur lors de la demande d’action.

•         Modelid : identifiant du modèle

•         Procid : identifiant de la procédure

•         Caseid : identifiant du cas

•         Transid : identifiant de la transaction

 

Pproctransform

•         Table associative entre ptrans et proctrans. Elle permet d’indiquer la suite logique des procédures. Lors de l’envoi d’un livrable, on récupère le procidsrc et procidtrg (source et targe) du livrable et on ajoute une ligne dans la table pproctransfrom. Le passage à l’étape suivante (procédure) se fera à partir de cette table. Elle permet également de faire « machine arrière » en cas de remise au chef ou retour pour non-conformité

•         Procid : identifiant de la procédure

•         Caseid : identifiant du cas

•         Transid : identifiant de la transaction

•         Refprocid : identifiant de la procédure qui amène à la procédure en cours (procid) = procédure parente

 

Tables de status (checkbox)

•         Prulesstatus, Ptaskstatus,Psystemstatus, Pinfostatus, Pdecisionstatus

•         Tables de correspondances entre le livrable et la transaction. (checkbox)

–        Caseid : identifiant du cas

–        Procid : identifiant de la procédure

–        Caseid : identifiant de la transaction

–        Delivid : identifiant du livrable

–        Mstate : état (coché ou non)

–        Mdate : date de changement

•         La version actuelle de Messenger ne permet pas de « décocher »

 

Pdelivstatus

•         La sélection d’un livrable est gérée de la même façon qu’une « checkbox » d’une autre table status. En sélectionnant un livrable, la table pdelivstatus est alimentée pour connaître le livrable sélectionné.

•         Caseid: identifiant du cas

•         Procid : identifiant de la procédure

•         Transid : identifiant de la transaction

•         Delivid : identifiant du livrable

•         Mstate : status (1 ou 0)

•         Mdated : (date de sélection)

 

Etapes lors de la création d’un nouveau cas d’un model m

•         Ajout dans la table pcase d’un nouveau cas avec identifiant séquentiel. (nouveau cas X)

–        Nom du cas sauvé

•         Ajout dans la table ptrans d’une nouvelle transaction (séquentielle) pour le cas X.

–        Description du cas sauvé

•         Récupération dans la table pproc du modelid = m et de procid sélectionné lors du start

•         Ajout dans la table pproctrans d’un nouvel enregistrement, avec le pprocid récupéré ci-dessus, le caseid et le transid.

 

•         Les livrables suivants sont affichés à l’aide de la table pdeliv qui contient la procédure source et destination. On récupère tous les delivid ayant comme procidsrc le procid récupéré auparavant.

 

•         Affichage des livrables sélectionnés

 

•         Lors de l’envoi d’un livrable sélectionné, on récupère le procidtrg (procédure cible), on alimente la table pdelivstatus avec le livrable sélectionné.

 

•         Lors de l’envoi d’un livrable on alimente également la table pproctransfrom avec la procédure source et destination.

 

•         Un nouvel enregistrement est créé dans pproctrans avec le procid récupéré depuis le livrable sélectionné (procidtrg). La table pproctransfrom permet de reconstituer la hiérarchie des procédures

 

•         Le bouton « terminer » modifie uniquement le status de la procédure en le mettant en terminé (procid = 98).

 

•         Les procédures avec le statusid à 98 ne sont plus affichées la boîte de travail.
